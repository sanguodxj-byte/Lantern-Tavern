# Lantern Tavern UI 编辑工作流与架构

这份文档是 UI 修改的默认工作协议。目标不是把所有 UI 一次性重写，而是让每次修改都只需要理解一个页面模块，并且能通过场景、脚本、测试和截图形成闭环。

## 1. UI 的四层结构

```text
游戏状态 / GameEvents / 各领域模块
            ↓ 业务适配
页面状态与用户意图（页面脚本）
            ↓ 生命周期
UiScreen / CanvasLayer UI 外壳
            ↓ 视觉实现
Godot 场景、Theme 变体、像素资源、动画
```

### 1.1 视觉实现层

场景文件（`.tscn`）只描述布局、节点层级、主题变体和默认资源。优先使用容器、锚点、`custom_minimum_size` 与主题资源，不在脚本中批量创建布局节点。

统一主题为 `scenes/ui/lantern_theme.tres`。新增控件先选择已有变体（如 `ScreenTitle`、`SectionTitle`、`PrimaryButton`、`SlotButton`、`HUDPanel`），只有确实出现新的语义状态时才新增变体，并同步主题测试。

### 1.2 UI 外壳层

页面型 UI 继承 `UiScreen`，获得 `open(payload)`、`close(result)`、`is_open()` 和 `request_navigation(route, payload)`。HUD、提示、受击闪烁等跨页面覆盖物继续使用 `CanvasLayer`，但不得把页面导航逻辑塞进 HUD。

`UiScreen` 是页面的生命周期 seam：页面可以在不创建完整世界场景的情况下被测试和预览。

### 1.3 页面状态与用户意图层

页面脚本负责：

- 把领域数据转换为标签、列表、图标和进度条需要的显示状态；
- 响应按钮、拖拽、键盘和鼠标输入；
- 发出“开始游戏、打开设置、返回”等用户意图。

页面脚本不负责散落的场景路径。场景切换统一经 `UiNavigation.navigate()`，路径登记在 `globals/ui/ui_route_catalog.gd`。

大面板迁移时，优先把“数据读取/格式化”提取成可独立测试的适配模块，再拆视觉子场景；不要先把一个 1200 行脚本机械切成多个薄包装。

### 1.4 领域状态层

UI 可以读取领域模块公开的只读状态或订阅事件，但不应该把装备、酿造、战斗规则重新实现一遍。修改业务规则时回到对应 `globals/` 或角色组件；修改呈现时留在 `scenes/ui/`。

## 2. 新增或修改页面的标准流程

### 第一步：确定页面类型

先判断这是页面、HUD、弹窗还是提示：

- 页面：继承 `UiScreen`，有明确的打开/关闭和路由关系；
- HUD：继承 `Control` 或 `CanvasLayer`，由世界生命周期挂载；
- 弹窗：作为页面的子场景，通过信号返回结果；
- 提示：复用 `InteractionHintBase` 或同类现有基类。

不要因为节点上有按钮就把 HUD 变成页面，也不要让页面直接扫描世界节点树寻找业务对象。

### 第二步：先画信息结构，再改节点

用以下顺序在场景树中安排节点：

1. `ScreenRoot`：负责页面范围、鼠标策略和主题；
2. `Header`：标题、返回、上下文操作；
3. `MainLayout`：页面主体的列/行容器；
4. `Content`：列表、预览、详情等语义区域；
5. `Footer`：确认、取消和操作提示。

节点名使用语义名，不使用 `Node2`、`LeftThing` 等位置名。脚本访问优先使用 `%UniqueName` 或稳定的短路径。

### 第三步：先做静态布局

先在编辑器中完成锚点、容器和最小尺寸，再接数据和动画。以 1920×1080 为基准检查，同时检查窗口缩放和中文/英文长度。固定像素 UI 必须保持 nearest 过滤、像素字体和现有主题契约。

### 第四步：接入生命周期与路由

页面脚本不要新增 `get_tree().change_scene_to_file()`。新增路径先登记到 `UiRouteCatalog`，再通过 `UiNavigation.navigate(route, payload)` 发起切换。页面打开时使用 `open(payload)`，关闭时通过 `close(result)` 返回结果。

### 第五步：接入数据和事件

采用“事件进入页面、状态单向流向节点、操作向领域模块发出”的方向：

```text
领域事件 → 页面适配 → render(state) → 场景节点
用户输入 → 页面意图 → UiNavigation / 领域模块
```

避免在 `_process()` 中无条件刷新所有控件。只有没有可用信号的外部状态才允许使用定时轮询，并且必须有脏标记或快照比较。

### 第六步：验证

每次 UI 修改至少完成：

1. 对应 gdUnit4 单元/场景测试；
2. 所有受影响页面 `PackedScene.instantiate()`；
3. 主题一致性测试；
4. 需要视觉判断时运行已有 UI 截图工具，并检查 PNG 不是空白或纯色；
5. 在中文和英文环境各检查一次文本溢出与焦点顺序。

## 3. 页面拆分规则

拆分前先问“删除这个模块后，复杂度是否会在多个调用处重新出现”：

- 如果只是转发属性或信号，删除它，避免浅模块；
- 如果它集中处理一组状态映射、交互规则或生命周期，它是深模块，保留独立 seam；
- 一个具体实现暂时只有一个适配器时，可以先保留函数 seam；出现第二种数据来源或预览模式时，再升级为明确 Adapter。

当前迁移顺序：

1. 页面生命周期与路由（已建立）；
2. 装备/模型详情的显示模型与格式化；
3. 酒馆装备面板的装备、技能、材料三个状态区域；
4. 战斗 HUD 的生命/护盾/Buff/日志四个呈现区域；
5. 跨页面弹窗、提示和输入焦点统一管理。
