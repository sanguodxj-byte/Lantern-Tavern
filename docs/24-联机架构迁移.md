## 1. 目标与非目标

- 支持 2–4 人合作游玩同一间酒馆、同一张地牢地图与同一场战斗。
- 保留现有单机入口、存档和本地测试，未开启联机时不依赖网络。
- 采用服务器权威：客户端只提交输入/意图，服务器决定移动、交互、掉落、伤害与持久化结果。
- 第一版不做匹配、账号、云存档、NAT 穿透或 MMO 级持久世界。

## 2. 当前架构的联机障碍（实证）

全仓搜 `multiplayer / rpc / @rpc / MultiplayerSynchronizer / peer_id / set_multiplayer_authority`：**游戏代码为零**（唯一命中是 gdUnit4 addon 自带 TCP，与游戏无关）。

玩家状态分布：

| 状态 | 载体 | 性质 |
|---|---|---|
| `current_player` | `GameState.current_player` | 全局唯一 Player 节点引用 |
| `expedition_inventory` | `GameState.expedition_inventory`（`.new()` 实例） | GameState 持有 |
| `equipment_loadout` | `GameState.equipment_loadout`（`.new()` 实例） | GameState 持有 |
| 属性/派生率 | `AttrPanel`（**autoload 单例**） | 全局唯一 |
| 技能状态 | `SkillRuntime`（**autoload 单例**） | 全局唯一 |

→ 隐含「世界里恰好一个玩家」假设，无法 per-peer。

## 3. 已落地：PlayerContext 句柄层（Phase 0）

**文件**：`globals/core/player_context.gd`（新增，`class_name PlayerContext extends RefCounted`）

聚合一个玩家的全部状态：

- `player_node: Node3D`
- `attributes: AttrPanel`（经 preload const 类型化，无 class_name 冲突风险）
- `skills: SkillRuntime`
- `inventory: ExpeditionInventory`
- `loadout: EquipmentLoadout`

**`GameState` 改造（纯加法，零行为变化）**：

- 新增 `var _player_context: PlayerContext = null`
- 新增 `func player_context() -> PlayerContext`：**惰性**创建并 `bind_to_globals()`，规避 autoload 初始化顺序问题
- `register_player()` 同步 `_player_context.player_node = player`

**两个工厂**：

- `bind_to_globals(player)` — 过渡期桥接，绑定到现有全局单例（行为=现状）
- `for_peer(attrs, sk, inv, lo, player)` — 未来联机工厂，由调用方传入已初始化的独立实例

**设计要点**：不新增 `class_name` 到 AttrPanel/SkillRuntime（避免与 autoload 名冲突）；用 preload const 做类型化；不改动任何现有调用点 → **单机行为完全不变**。

## 5. 后续分阶段路线

### Phase 0.5 — 网络传输与 peer 注册层（已落地 ✅）
- **文件**：`globals/core/network_manager.gd`（新增 autoload）。
- Godot 4 原生 `SceneMultiplayer` + `ENetMultiplayerPeer`，服务器权威模型（host = peer 1）。
- `host(port)` / `join(address, port)` / `disconnect_session()`；信号 `peer_joined` / `peer_left` / `connected_to_server` / `connection_failed` / `server_disconnected` / `lobby_updated`。
- `peer_contexts: Dictionary`（peer_id → PlayerContext）登记位 + `register_peer_context` / `get_peer_context` / `connected_peers`。
- `request_player_spawn(peer_id)` RPC 骨架（`@rpc("any_peer","call_remote","reliable")`），当前 no-op，单机永不触发；是 Phase3 玩家生成的接入点。
- 客户端 join() 在握手完成前不伪造本地 peer id；连接失败/服务器断开会清理底层 peer 与会话元数据。
- equest_player_spawn() 使用远端 sender id，不信任客户端传入的 peer_id。
- **关键安全属性**：单机下 host()/join() 永不被调用 → 本单例完全惰性，**零行为变化**。
- 见 `TaskList`（#21 完成；#22–#27 为后续 Phase）。

## 4. 目标运行时架构

`	ext
MainMenu / Lobby -> NetworkManager -> SessionRoot
SessionRoot: PlayerRegistry / WorldState / CommandRouter / AuthoritySystems
Replication: MultiplayerSpawner + MultiplayerSynchronizer
Client: camera / HUD / VFX / audio / interpolation
` 

权威边界：客户端提交移动、交互、攻击、技能和空间切换意图；服务器校验距离、视线、冷却、权限、命中、伤害、掉落、库存和存档；客户端只呈现服务器结果。

同步策略：移动 20–30 Hz 快照并插值；战斗/交互/库存使用可靠 RPC 与序列号去重；地牢同步 seed、生成配置版本和实体快照；空间切换使用 world_revision，revision 不一致时请求完整快照。

## 5. 分阶段路线

### Phase 1 — AttrPanel / SkillRuntime 可实例化（前置，无联机也可做）
- 为两脚本加 `class_name`（需先确认不与 autoload 名冲突；或改用 preload const 工厂）。
- 把依赖 `_ready` 的初始化抽成显式 `init_*()`，使 `.new()` 后不依赖入树即可用。
- 风险：改动面广，需 Godot 二进制跑测试确认（本环境无，待编辑器/CI 验证）。

### Phase 2 — 调用点迁移到上下文
- 全局替换 `AttrPanel.xxx` → `GameState.player_context().attributes.xxx`
- 全局替换 `SkillRuntime.xxx` → `GameState.player_context().skills.xxx`
- `player_node` / `inventory` / `loadout` 同理经上下文访问。
- 此步完成 = 单机已完全 per-player 化，仍是单机单实例。

### Phase 3 — 联机运行时
- 引入 `MultiplayerSpawner` / `MultiplayerSynchronizer` 复制敌人、掉落、门、宝箱。
- 伤害结算改服务器权威 + RPC 下发（现有 `damage_resolver` 为本地同步，无 authority/复制钩子）。
- 输入按 `is_multiplayer_authority()` 门控。
- 每个 peer `PlayerContext.for_peer(...)` 各自独立状态。
- 主机生成地牢 → 同步种子/布局给客户端。

- 越晚做 Phase 1–2，散在单例的调用点藤蔓越广，成本指数上升——建议单机稳定后尽早排期 Phase 1。

## 6. 第一版消息契约

所有客户端命令包含 `protocol_version`、`world_revision`、`client_tick`、`sequence`；服务器事件包含 `world_revision`、`server_tick`、`sequence`。

推荐命令：`join_session`、`request_spawn`、`input_frame`、`request_interact`、`request_attack`、`request_skill`、`request_space_transition`、`request_save`。

推荐事件：`session_ready`、`player_spawned`、`player_snapshot`、`entity_spawned`、`entity_despawned`、`combat_resolved`、`inventory_changed`、`space_snapshot`、`command_rejected`。

## 7. 验收标准

- 单机不调用 `host()`/`join()` 时，现有 gdUnit4 测试通过，且 `NetworkManager` 不创建 socket。
- 两个本地进程完成 host → join → 双方生成角色 → 移动同步 → 交互 → 伤害结算 → 断线清理。
- 客户端篡改 `peer_id`、越界交互目标、重复命令、错误 `world_revision` 时，服务器拒绝且不改变权威状态。
- 地牢布局在相同种子/配置版本下可重建；不同版本握手失败而不是静默产生不同地图。
- 服务器重启/客户端重连不会写入或读取其他 peer 的角色状态。
