## 1. 总体结论

当前项目适合采用：

- **Godot 4.7 `SceneMultiplayer` + ENet + 服务器权威 + Listen Server 起步**，后续可切换 Dedicated Server。

**不建议直接把现有单机逻辑简单加上 `@rpc`。** 当前项目的核心问题不是缺少网络 API，而是存在大量“全局只有一个玩家”的假设：

- `GameState.current_player`
- `GameState.expedition_inventory`
- `GameState.equipment_loadout`
- `AttrPanel` autoload
- `SkillRuntime` autoload
- 敌人默认追踪 `GameState.current_player`
- 地牢、酒馆、装备面板和存档系统默认绑定单一玩家
- 伤害结算和拾取逻辑目前由本地节点直接执行

因此需要分成两条线并行推进：

1. **状态架构重构**：将玩家状态改成 per-peer。
2. **网络运行时接入**：由服务器控制共享世界和所有权威结果。

现有联机基础代码位于：

- `globals/core/network_manager.gd`
- `globals/core/player_context.gd`
- `docs/24-联机架构迁移.md`

## 2. 推荐网络模型

### 2.1 第一阶段：Listen Server

玩家一创建房间，就同时承担：

- 游戏服务器
- 本地客户端
- 世界状态持有者
- 战斗和掉落权威

结构如下：

```
Host Process
├── Godot SceneTree
├── NetworkManager
├── SessionRoot
├── World
├── Server Authority Systems
├── Host Player
└── Host Client Presentation

Client Process
├── Godot SceneTree
├── NetworkManager
├── SessionRoot Proxy
├── World Proxy
└── Client Presentation
```

**优点**：

- 开发成本低
- 可以直接复用 Godot 场景
- 不需要一开始维护独立服务器程序
- 方便本地双进程测试

**缺点**：

- 主机退出时房间结束
- 主机拥有较高作弊能力
- 主机性能会影响其他玩家

### 2.2 第二阶段：Dedicated Server

后续将服务器逻辑拆成无 UI 的专用运行模式：

```
Dedicated Server
├── NetworkManager
├── SessionRoot
├── WorldState
├── PlayerRegistry
├── CombatAuthority
├── LootAuthority
└── SaveAuthority
```

服务器**不加载**：

- 摄像机
- HUD
- 第一人称视图模型
- 本地音频
- 仅用于表现的粒子和后处理
- 酒馆展示型 UI

**建议**：一开始就让游戏规则代码不依赖 `World` 的 UI 节点，这样后续可以平滑切换 Dedicated Server。

## 3. 网络层分层设计

### 3.1 NetworkManager

**职责限制为**：

- 创建和关闭 ENet peer
- Host / Join
- 连接、断线、peer 生命周期
- 网络协议版本握手
- 传输错误处理
- RPC 基础封装
- peer ID 管理

**不应负责**：

- 生成玩家
- 计算伤害
- 修改库存
- 生成地牢
- 保存游戏
- 管理战斗状态

**文件**：`globals/core/network_manager.gd`

### 3.2 SessionRoot

新增场景级节点：

```
SessionRoot
├── PlayerRegistry
├── WorldState
├── CommandRouter
├── SnapshotReplicator
├── CombatAuthority
├── InteractionAuthority
├── LootAuthority

└── SaveAuthority
```

**职责**：

- 管理当前多人会话
- 保存当前世界 revision
- 注册和注销玩家
- 处理客户端命令
- 调度服务器权威系统
- 管理场景空间切换
- 对客户端发送快照和事件

**建议新增目录**：

```
globals/multiplayer/
├── session_root.gd
├── player_registry.gd
├── world_state.gd
├── command_router.gd
├── snapshot_replicator.gd
├── combat_authority.gd
├── interaction_authority.gd
├── loot_authority.gd
├── tavern_authority.gd
└── save_authority.gd
```

### 3.3 PlayerRegistry

维护：

- `Dictionary[int, PlayerContext]`
- `Dictionary[int, Player]`
- `Dictionary[int, PlayerSession]`

示例：

```
peer_id 1
├── Player node
├── PlayerContext
├── connection state
├── last command sequence
├── last input tick
├── current world revision
└── spawned state

peer_id 2
├── Player node
├── PlayerContext
├── connection state
├── last command sequence
├── last input tick
├── current world revision
└── spawned state
```

所有服务器逻辑都通过：

- `PlayerRegistry.get_context(peer_id)`
- `PlayerRegistry.get_player(peer_id)`

访问玩家，而不是使用：

- `GameState.current_player`
- `AttrPanel`
- `SkillRuntime`

## 4. 玩家状态重构

### 4.1 PlayerContext 作为唯一玩家状态入口

当前已有：`globals/core/player_context.gd`

**目标结构**：

```gdscript
class_name PlayerContext
extends RefCounted

var peer_id: int
var player_node: Player

var attributes: PlayerAttributes
var skills: PlayerSkills
var inventory: ExpeditionInventory
var loadout: EquipmentLoadout
var progression: PlayerProgression
var session_stats: PlayerSessionStats
```

**建议继续加入**：

```gdscript
var peer_id: int = 0
var display_name: String = ""
var is_ready: bool = false
var is_alive: bool = true
var current_space: String = ""
var current_world_revision: int = 0
var last_command_sequence: int = 0
var last_input_tick: int = 0
```

### 4.2 单机兼容策略

单机模式下仍然保留旧访问接口：

- `GameState.current_player`
- `GameState.expedition_inventory`
- `GameState.equipment_loadout`

但这些接口内部应委托给：

- `GameState.player_context()`

迁移顺序：

```
旧单机调用
    ↓
GameState 兼容代理
    ↓
PlayerContext
    ↓
真正 per-peer 状态
```

单机不应该因此产生行为变化。

### 4.3 AttrPanel 和 SkillRuntime

当前二者是 autoload 单例，这是联机的主要阻塞点。

**目标**：

- 单机：`GameState.player_context().attributes` / `GameState.player_context().skills`
- 联机：`PlayerRegistry.get_context(peer_id).attributes` / `PlayerRegistry.get_context(peer_id).skills`

**迁移后**：

- `AttrPanel` 可以是普通 `RefCounted` 或 `Node`
- `SkillRuntime` 可以是普通 `RefCounted` 或 `Node`
- autoload 只作为单机兼容代理保留
- **不允许**服务器通过全局 `AttrPanel` 计算所有玩家属性

**建议最终命名**：

- `PlayerAttributes`
- `PlayerSkillRuntime`

如果为了减少现有改动继续使用旧脚本名，也必须保证它们可以 `.new()` 创建多个实例。

## 5. 权威边界

### 5.1 总原则

**客户端只发送**：意图、输入、请求

**服务器决定**：是否允许、实际结果、最终状态

**客户端不能直接决定**：

- 自己的位置
- 自己的生命值
- 敌人的生命值
- 伤害数值
- 掉落内容
- 背包数量
- 金币和材料
- 宝箱是否打开
- 门是否开启
- 地牢出口是否完成
- 酿造是否完成
- 存档内容

### 5.2 权威矩阵

| 系统 | 客户端 | 服务器 |
|---|---|---|
| 移动 | 发送输入方向、跳跃、朝向 | 计算碰撞和最终位置 |
| 摄像机 | 完全本地 | 不同步 |
| 动画 | 根据服务器状态表现 | 决定状态标签 |
| 交互 | 发送目标和意图 | 验证距离、视线、权限、冷却 |
| 近战攻击 | 发送攻击意图 | 计算命中、伤害、击退 |
| 远程攻击 | 发送瞄准和发射意图 | 生成投射物并计算碰撞 |
| 技能 | 发送技能请求 | 验证资源、冷却、目标和结果 |
| 敌人 AI | 不运行权威 AI | 服务器运行 |
| 敌人动画 | 本地表现 | 同步状态 |
| 掉落 | 不决定掉落 | 服务器生成随机结果 |
| 拾取 | 发送拾取请求 | 验证距离并修改库存 |
| 装备 | 发送装备请求 | 验证并写入 loadout |
| 酒馆经营 | （单人本地，**不在联机范围**） | 玩家只继承各自存档状态进入地牢；酿造/升级/共享经济不联网 |
| 地牢生成 | 可本地重建视觉 | 服务器决定 seed、配置版本和实体 |
| 存档 | 请求保存 | 服务器唯一写入 |

## 6. 移动同步方案

### 6.1 第一版不要直接同步客户端位置

**不推荐**：

```
客户端移动 → 直接把 position 发给服务器 → 服务器广播 position
```

这会导致：

- 穿墙
- 速度作弊
- 瞬移
- 穿过敌人
- 不一致碰撞

### 6.2 推荐第一版：服务器采样输入

**客户端发送输入帧**：

```json
{
    "protocol_version": 1,
    "world_revision": 12,
    "client_tick": 10450,
    "sequence": 900,
    "move": [0.0, -1.0],
    "look_yaw": 1.2,
    "look_pitch": -0.1,
    "jump": false,
    "sprint": true
}
```

**服务器**：

- 验证 peer 是否已生成角色
- 验证 `world_revision`
- 验证 `sequence` 是否重复
- 限制输入范围
- 根据服务器物理状态执行移动
- 生成最终位置和速度
- 广播状态快照

**服务器状态**：

```json
{
    "peer_id": 2,
    "server_tick": 12022,
    "position": [x, y, z],
    "velocity": [x, y, z],
    "rotation_y": 1.3,
    "movement_state": "running",
    "grounded": true
}
```

### 6.3 客户端表现

第一版使用：

- 本地玩家：短暂预测 + 服务器纠正
- 远端玩家：插值
- 不做完整 rollback
- 不允许客户端长期偏离服务器

**推荐参数**：

- 服务器模拟：30 Hz
- 输入发送：20–30 Hz
- 状态广播：15–20 Hz
- 客户端插值缓冲：100–150 ms
- 位置纠正阈值：0.15–0.30 m

## 7. 战斗同步方案

### 7.1 服务器权威伤害

现有战斗核心：

- `globals/combat/combat_engine.gd`
- `globals/combat/damage_resolver.gd`
- `globals/combat/combat_bridge.gd`

这些文件中的纯数值计算可以复用，但调用入口需要改变。

**当前大致流程**：

```
Player 本地输入 → 本地命中检测 → 本地 CombatEngine → 本地扣血
```

**目标流程**：

```
客户端攻击输入 → request_attack RPC → CombatAuthority → 服务器命中检测
→ DamageResolver → 服务器扣血 → combat_resolved 事件广播
```

### 7.2 攻击命令

```json
{
    "world_revision": 12,
    "client_tick": 200,
    "sequence": 80,
    "attack_type": "melee",
    "weapon_slot": 0,
    "origin": [x, y, z],
    "direction": [x, y, z],
    "charge_ratio": 0.75,
    "target_hint": 4312
}
```

**注意**：

- `origin` 只能作为提示，不能完全信任
- 服务器应从玩家节点重新读取位置
- `target_hint` 只能用于优化，不可直接作为命中结果
- 武器、属性和技能必须从服务器的 `PlayerContext` 读取

### 7.3 服务器攻击验证

服务器检查：

- 玩家是否存在
- 玩家是否存活
- 玩家是否处于允许攻击状态
- 武器槽位是否有效
- 武器是否属于该玩家
- 攻击冷却是否结束
- 技能资源是否足够
- 目标是否在合法范围
- 是否符合视线、角度和碰撞规则
- `world_revision` 是否正确
- `sequence` 是否重复

然后调用：

- `DamageResolver.resolve_attack(...)`

服务器应用：

- 当前生命值
- 击退
- 受击状态
- 眩晕
- 死亡
- 掉落
- 战斗日志事件

**客户端只接收**：

```json
{
    "event": "combat_resolved",
    "attacker_peer_id": 2,
    "defender_entity_id": 4312,
    "damage": 18,
    "critical": true,
    "knockback": [x, y, z],
    "stagger_seconds": 0.25,
    "defender_life": 42
}
```

- `velocity`

- `current_life`

## 10. 酒馆边界

酒馆完全属于单人本地玩法，不进入多人会话，不由 `NetworkManager`、`SessionRoot` 或任何 multiplayer authority 管理。

多人会话只在玩家进入地牢（出征）后建立。进入地牢时，服务器接收并校验每个玩家的单人存档摘要，创建独立的 `PlayerContext`；酒馆状态不会作为共享多人状态同步。

以下内容明确不联网：

- 酒馆场景和酒馆内节点
- 酒馆金币、材料仓库、符文仓库
- 酿造、发酵和生产队列
- 酒馆升级、声望、顾客和员工
- 酒馆日期、经营收益和本地 UI

### 10.1 地牢玩家私有状态

每个玩家独立：

- 装备
- 个人属性
- 技能
- 个人携带背包
- 个人生命值
- 个人出征状态
- 进入地牢时从单人存档继承的摘要

### 10.2 地牢共享交互冲突

例如两个玩家同时拿取同一个地牢材料：

```
客户端 A request_pickup(item_id)
客户端 B request_pickup(item_id)
```

服务器串行处理：

```
A 成功
B 返回 item_already_taken
```

客户端不能通过本地显示判断物品是否仍可拾取。所有多人交互必须是地牢内实体交互，不经过酒馆系统。

## 11. 交互和拾取

所有交互统一进入：

```
CommandRouter → InteractionAuthority
```

建议交互命令：

```json
{
    "world_revision": 12,
    "client_tick": 400,
    "sequence": 123,
    "target_entity_id": 9001,
    "interaction_type": "pickup"
}
```

**服务器验证**：

- entity 是否存在
- entity 是否属于当前空间
- 玩家距离
- 玩家视线
- 玩家是否存活
- 目标是否已被其他玩家消费
- 当前交互类型是否允许
- 是否有冷却或权限要求

**交互结果**：

```json
{
    "event": "interaction_result",
    "sequence": 123,
    "success": true,
    "target_entity_id": 9001,
    "inventory_delta": {
        "goblin_tooth": 1
    }
}
```

## 13. 断线和重连

### 13.1 断线处理

玩家断线后：

- 从在线玩家列表移除
- 暂停或销毁玩家实体
- 释放交互锁
- 释放正在进行的操作
- 保留玩家私有状态一段时间
- 广播 `player_despawned`
- 超时后清理 `PlayerContext`

**建议**：

- 断线保留：60 秒
- 重连 token 有效期：60 秒
- 房主断线：第一版直接结束房间

### 13.2 重连快照

客户端重连后不能只接收增量事件，应先获取：

```
session_snapshot
├── protocol_version
├── world_revision
├── current_space
├── tavern_state
├── player_context
├── player_entities
├── enemies
├── loot
├── doors
├── chests
└── expedition_state
```

然后再接收增量事件。

- 地牢静态几何使用 seed 重建

- 远距离实体降低同步频率

- 服务器不保存客户端传来的完整状态

**建议同步优先级**：

- **高频**：玩家位置、速度、朝向、状态
- **中频**：敌人位置、目标、战斗状态
- **低频**：宝箱、门、掉落、酒馆状态
