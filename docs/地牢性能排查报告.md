# 地牢（Dungeon）性能综合排查报告

> 范围：地牢探险运行时（`scenes/expedition/`、`globals/dungeon/`、`globals/multiplayer/`、`globals/core/network_manager.gd`、`scenes/characters/enemies/`、`scenes/props/torch/`）
> 引擎：Godot 4.7 + GDScript，桌面端 Forward+ / 移动端 forward_mobile
> 排查方式：代码静态走查（已读关键文件与行号佐证），非线上 profiling

---

## 一、结论摘要

地牢的整体架构**已经有不少有效优化**（遮挡剔除、地形 MultiMesh 实例化、灯光预算=12、敌人 imposter LOD、动画冻结、物理体按 chunk 冻结、F3 性能 HUD）。因此**显存/overdraw/静态开销基本可控**。

真正的瓶颈集中在两类：

1. **生成/进场阶段是一次性重活** —— 整张地牢同步生成 + 全图敌人一次性实例化，造成进场卡顿与显存峰值（尤其多敌人蒙皮 rig）。
2. **高频实时路径缺少节流** —— 实体状态同步走的是**无节流的 reliable RPC**（对比玩家快照已是 30Hz 节流），密集战斗会推高消息率与 CPU。

次要瓶颈：流控跨块全表扫描、每敌逐帧 LOS 射线、火把常驻粒子+常播 3D 音（不受灯光预算约束）。

## 三、发现明细（按严重度）

### 🔴 P0 — 整图同步生成 + 全图敌人一次性实例化

**证据**
- `scenes/expedition/procedural_dungeon.gd` `_ready()`（~L75）：串行执行 `DungeonGenerator.generate → validate → ConnectivityValidator → HazardPlanner → SpawnPlanner → DungeonSceneBuilder.build → streaming_controller → DungeonRuntime.start`，**整张地牢在加载帧一次性生成**，无异步/分帧。
- `scenes/expedition/dungeon_runtime.gd:160` `spawn_enemies()` → `:174` `spawner.spawn_enemies_from_layout(layout, spawn_root, player_node)` —— **一次性实例化全图敌人**。
- `globals/dungeon/dungeon_spawner.gd:200` `spawn_enemies_from_layout` 遍历所有房间，每房 `clamp(1, round(positions*count_per_room*0.25), 5)`，全图敌人上限约 **20–40**。
- 每个敌人（`scenes/characters/enemies/enemy.gd`）带完整 `Skeleton3D` + `PhysicalBoneSimulator3D` + `AnimationPlayer` + `NavigationAgent3D` + 常驻 `VoxelRagdoll`，且 `@onready` 在 `_ready` 创建。

**影响**
- 进场/读档瞬间主线程长停顿（生成 + 实例化数十个蒙皮 rig）。
- 显存峰值高；headless 下重复实例化 skeleton-rig 敌人易崩溃（见沙箱记忆）。
- 这些敌人随后才被 stream 控制器按 chunk 冻结，但**实例化本身已发生**，冻结只省运行时不省加载。

**建议**
- **分房/按 chunk 懒加载**：不在 `spawn_enemies` 一次性实例化，改为随玩家推进、由 `DungeonStreamingController` 在进入特定 chunk 半径时才实例化该房间的敌人（与现有视觉/物理流控同构）。
- **对象池 + 分帧**：若必须预生成，用 `call_deferred` 分批（每帧 N 个）实例化，避免单帧卡死。
- 复用：死亡/离场敌人的 rig 不立即 `queue_free`，归还对象池供下一波复用。

### 🟠 P1 — 实体状态同步无节流 + reliable 广播

**证据**
- `globals/core/network_manager.gd:441` `server_update_entity(entity_id, patch)` → `session.update_entity` → **立即** `_dispatch_event(res["event"], 0)`（:447）。
- `_dispatch_event`（:312）最终 `rpc_server_event.rpc(event)`（:318）。
- `rpc_server_event` 声明为 **`@rpc("authority", "call_remote", "reliable")`**（:554）—— 即**每次实体状态变更都发一条可靠 RPC，无任何节流/合并**。
- 对照 `player_snapshot` 走的是 `_flush_snapshots`（:356），按 `SNAPSHOT_BROADCAST_HZ=30` 节流、每 peer 只缓冲最新一帧（:360）。实体同步**没有**这套节流。

**影响**
- 战斗中 HP/位置/状态频繁变化 → 高频 reliable 消息。reliable 有重传与排序保证，会放大带宽与 CPU，且在丢包时阻塞后续消息（队头阻塞风险）。
- 多人联机时 Net msg/s（F3 HUD 可见）会显著抬升。

**建议**
- 把实体状态更新并入 30Hz 节流缓冲：高频字段（位置/HP）按 `SNAPSHOT_BROADCAST_HZ` 合并后下发，而非每次 `server_update_entity` 即时发。
- 对非关键高频字段改用 `unreliable` + 客户端插值（位置可用 unreliable；HP/状态变更仍可 reliable，但应批量化）。
- 参考 `EntitySyncAuthority.build_delta` 做差量，只下发变化字段。

### 🟠 P1 — 流控跨块全表扫描 O(N)

**证据**
- `dungeon_streaming_controller.gd:137` `_update_lights()`：**先把 `_light_chunks` 全部灯光 `visible=false`（全扫），再对半径 2 chunk 内灯光排序取前 12 激活**（:138–154）。
- `_update_visuals()`（:179）/ `_update_terrain()`（:192）：遍历**所有**已注册 chunk `keys()` 设可见性。
- 这些只在玩家跨 chunk 时触发（每 24m 移动一次），但大地图一次性全扫代价随地牢规模线性增长。

**影响**
- 大地图、跨块瞬间出现掉帧尖峰（尤其灯光先全隐再排序）。
- 重复处理未变化的 chunk（只需处理差集）。

**建议**
- 维护"上一激活 chunk 集合"，只对新进入/新离开的 chunk 做增删，避免全表清扫。
- 灯光预算维护一个"已激活灯光集合"，跨块时只移除离开半径的、补充进入半径且尚未激活的（仍按距离取舍到 12）。
- 用网格/空间哈希索引 chunk，避免 `keys()` 全遍历。

### 🟠 P2 — 每敌逐帧 LOS 射线检测

**证据**
- `scenes/characters/enemies/enemy.gd` `_tick_detection`（~L330 附近）调用 `has_line_of_sight_to(target)`（:383），后者做物理射线（`PhysicsDirectSpaceState3D.intersect_ray`）。
- 追击态/索敌态敌人各自每帧一次 raycast；密集战斗时数十次 raycast/帧。

**影响**
- 大量同步射线会占用物理线程/主线程，叠加在 P0 敌人数量上更明显。

**建议**
- 检测频率节流：按 `RENDER_OPT_INTERVAL`（0.2s，已在 `enemy.gd` 用于渲染优化）节奏做 LOS，而非每帧。
- 已冻结（离屏/远处）敌人不进入检测分支（现有流控已 `set_physics_process(false)`，需确认检测也在该分支内）。
- 结果缓存：相邻帧复用上一有效 LOS，仅在目标移动明显时重测。

---

### 🟠 P2 — 火把常驻粒子 + 常播 3D 音频（不受灯光预算约束）

**证据**
- `scenes/props/torch/torch.tscn`：每个火把含 `GPUParticles3D`（`amount=10, fixed_fps=15, visibility_range_end=35, autoplay=true`，:48–57）**持续发射**，以及 `AudioStreamPlayer3D`（`autoplay=true`，:59–64）常播火焰循环音。
- 该 tsch **不含任何 OmniLight3D**（真实灯来自装饰/陷阱/玩家，由灯光预算 12 控制）。所以**粒子的可见性只由 `visibility_range_end=35` 决定，不受 12 灯预算约束**。

**影响**
- 视野内多个火把 = 多个常驻 GPU 粒子系统 + 多个常播 3D 音源同时运行，开销随可见火把数线性增长。
- 音频多路常播在低端机/移动端尤其敏感。

**建议**
- 粒子发射与灯光预算联动：当火把灯光被预算关闭（`visible=false`）时，同步 `emitting=false`，避免"看不见的火仍在烧粒子"。
- 音频：远处/不可见火把暂停 `AudioStreamPlayer3D`；或同一区域共享一个环境火焰音，而非每把火一把。
- 用 `VisibilityEnabler3D` 的 `pause_particles/audio` 自动行为兜底。

---

### 🟢 P3 — 无全局 GI/SSAO（低优先）

- `project.godot` 未配置 SDFGI/VoxelGI/SSAO/SSR；环境光靠 `procedural_dungeon._setup_zone_ambient` 现建 `WorldEnvironment`（雾 + 顶光 + `DirectionalLight3D` shadow=false）。
- 视觉依赖动态点光源堆砌，已被 12 灯预算缓解。**不建议为"提亮"引入 SDFGI**（Forward+ 下 SDFGI 本身有显著成本），除非做专项评估。

## 四、度量方法（直接复用现有 F3 HUD）

`globals/perf/perf_monitor.gd` 已暴露：FPS、帧耗时、`RENDER_TOTAL_OBJECTS_IN_FRAME`、`RENDER_TOTAL_PRIMITIVES_IN_FRAME`、网络 msg/s（来自 `NetworkManager.get_net_stats()`）。

**复现与定位步骤**
1. 进场/读档时观察帧耗时尖峰 → 对应 P0（整图生成 + 全图敌人实例化）。
2. 多人战斗时按 F3 看 Net msg/s → 对应 P1（实体同步无节流）。
3. 大地图快速移动（跨 chunk）时观察帧耗时抖动 → 对应 P1 流控全扫。
4. 密集敌群 + 多火把场景观察帧耗时与音频卡顿 → 对应 P2 射线/粒子/音频。

**建议补充的 HUD 指标（可选，便于量化上述热点）**
- 当前激活动态灯光数 / 粒子系统数 / 激活实体数 / 流控 chunk 数。
- 每帧 raycast 次数（可临时打点 `_tick_detection`）。

## 实施状态（2026-07-21 已落地）

以下 5 项全部实现并通过 headless 解析校验：

| 项 | 文件 | 关键改动 |
|---|---|---|
| 流控差集增量 | `scenes/expedition/dungeon_streaming_controller.gd` | `_update_lights` 用 `_active_light_set` 差集（去掉全表 hide-all）；`_update_visuals/_update_terrain` 用 last-active 集合增量增删 |

| 敌人 LOS 节流 | `scenes/characters/enemies/enemy.gd` | `has_line_of_sight_to` 加 `_los_cache_timer`(LOS_INTERVAL=0.2s) 缓存射线结果 |
| P0 敌人分帧实例化 | `globals/dungeon/dungeon_spawner.gd` + `scenes/expedition/dungeon_runtime.gd` | 新增 `build_enemy_spawn_plan`/`instantiate_enemy_descriptor`；`spawn_enemies_from_layout` 加 `batched` 参数（默认 false 保同步契约）；`spawn_enemies` 取计划后按帧批量(4/帧)实例化，无场景树回退同步 |
| P1 实体同步 30Hz 节流 | `globals/core/network_manager.gd` | `server_update_entity` 有真实 peer 时把 `EVT_ENTITY_SNAPSHOT` 入 `_entity_update_buffer`(按 entity_id 合并)，随 `_flush_snapshots` 30Hz 下发；spawn/despawn 仍可靠即时 |
