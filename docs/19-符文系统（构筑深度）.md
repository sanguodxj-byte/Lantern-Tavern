# 19-符文系统（构筑深度）/ Rune System (Build Depth)

> 本文档是 **构筑深度（Build Depth）** 体系的核心独立文档。
> 它描述「符文（Rune）」——可镶嵌到技能槽、对技能数值与机制做乘法/加法修饰的模块化系统。
> 与之同名但**完全不同**的「符石（Escape Rune / 逃脱符石）」是探索撤离消耗品，见 `04-探索与地图生成.md`，不在本文档范围内（详见 §0）。

---

## §0 术语澄清：符石 ≠ 符文

项目里有两个容易混淆的概念，必须分开：

| 名称 | 英文名 | 本质 | 所在文档 | 是否构筑深度 |
|---|---|---|---|---|
| **符石** | Escape Rune / 逃脱符石 | 一次性消耗品：原地开启逃脱传送门，带 2.5s（正常/重荷）或 4s（超重）**读条** | `04-探索与地图生成.md` §3 | 否（探索/撤离道具） |
| **符文** | Rune | 可镶嵌进技能槽的**修饰器**，改变技能数值与机制标签 | 本文档 | **是（构筑深度核心）** |

代码侧已统一用「符文」：
- 数据：`globals/combat/rune_data.gd`
- 资产图标：`assets/textures/icons/runes/`（ember / quick / force / surge / launch / echo / guardian 共 7 个）
- 槽位运行时：`globals/combat/skill_runtime.gd`

> **设计待决**：逃脱符石（04）目前是一套独立掉落/读条逻辑，未复用本符文池。是否让两者共享 loot 来源、或让「逃脱符石」也走 `roll_rune` 权重，是后续需要拍板的点（见 §8 缺口 D8）。

## §1 系统定位：构筑深度的第三根支柱

游戏的构筑深度由三层相乘构成：

```
构筑深度 = 技能(15)  ×  装备(06)  ×  符文(本文档)  →  实时战斗(05)
```

- **技能（15）**：决定「能做什么」（踢击 / 冲撞 / 抓取投掷 / 滑铲 / 战术滑步 / 双轨领悟被动）。
- **装备（06）**：决定「基础面板与武器流派被动」。
- **符文（19）**：决定「同一技能如何被放大/变形」——是乘数层，也是玩家表达流派个性的主战场。

符文不新增技能，而是**放大差异**：同一个「冲撞」可以走「冲量位移流（surge×3）」或保持均衡；同一个「踢击」可以走「抛掷踢飞流（launch×3）」或「爆发响应式（ember+quick）」。这正是构筑深度的来源。

## §2 系统架构（实现方式，按层）

```
┌──────────────────────────────────────────────────────────────┐
│ L6 获取 / 掉落层                                               │
│   RuneData.roll_rune(source)  →  chest/elite/boss 加权随机     │
├──────────────────────────────────────────────────────────────┤
│ L5 持久化层                                                     │
│   SkillRuntime.serialize()/deserialize() → "slot_runes" 存读档  │
├──────────────────────────────────────────────────────────────┤
│ L4 UI 层                                                        │
│   RuneSlotDropButton (拖拽接收) → EquipmentPanel 槽位插座        │
├──────────────────────────────────────────────────────────────┤
│ L3 槽位运行时层 (autoload: SkillRuntime)                        │
│   7 槽位 · socket_rune/unsocket_rune · get_rune_capacity        │
│   get_effective_slot_skill(slot) → RuneData.apply_runes(...)    │
├──────────────────────────────────────────────────────────────┤
│ L2 应用层                                                       │
│   RuneData.apply_runes(skill, rune_ids)                         │
│     · 数值键 mul/add 修饰  → 写回 skill                        │
│     · 机制标签             → 合并进 skill["rune_effects"]       │
├──────────────────────────────────────────────────────────────┤
│ L1 数据层                                                       │
│   RuneData.RUNES (7 种) · SOURCE_WEIGHTS (掉落权重)             │
└──────────────────────────────────────────────────────────────┘
```

### L1 数据层 — `globals/combat/rune_data.gd`
- `const RUNES: Dictionary`：7 种符文定义，每项为 `{id, name, runic_name, rarity, mods, mechanics, desc}`。
- `const SOURCE_WEIGHTS: Dictionary`：按来源（chest / elite / boss）的掉落权重表。
- 静态访问器：`get_rune(id)`、`has_rune(id)`、`get_all_rune_ids()`、`get_rune_name(id)`、`apply_runes(skill, rune_ids)`、`roll_rune(source)`。

### L2 应用层 — `apply_runes(skill, rune_ids)`
- 对 `skill` 做 `duplicate(true)`，逐符文叠加：
  - **数值修饰**（`mods`）：键支持 `{"mul": x}` 或 `{"add": y}`（可并存，先乘后加）。
    - 若目标值是字典（如嵌套数值），则递归到其数值子键；若是标量 `int/float`，直接修饰。
  - **机制标签**（`mechanics`）：整体合并进 `skill["rune_effects"]`，供战斗层读取（如 `burn_chance`、`extra_projectiles`、`quickened`、`passive_guard`）。
- 结果写回 `skill["rune_ids"]`（已镶嵌列表）与 `skill["rune_effects"]`。

### L3 槽位运行时层 — `globals/combat/skill_runtime.gd`（autoload `SkillRuntime`）
- **槽位结构**：`TOTAL_SLOTS = 7`
  - 槽 0：F 键主动（动作技能，旧名 `SLOT_F_ACTION`）
  - 槽 1：G 键主动（武器主动，旧名 `SLOT_G_WEAPON`）
  - 槽 2–6：5 个被动槽（`SLOT_PASSIVE_1..5`）
- **容量**：`MAX_RUNES_PER_SLOT = 3`（主动槽） / `MAX_RUNES_PER_PASSIVE_SLOT = 2`（被动槽）。通过 `get_rune_capacity(slot)` 区分。
- **API**：`socket_rune(slot, rune_id)`（容量/存在性/已绑技能三重校验）、`unsocket_rune(slot, rune_index)`、`get_slot_runes(slot)`、`get_effective_slot_skill(slot)` → 内部调用 `apply_runes`。
- 默认 F 槽绑定 `踢击`，保证进地牢即可用。

### L4 UI 层 — `scenes/ui/rune_slot_drop_button.gd`
- `RuneSlotDropButton`（继承 `Button`），挂在装备面板的符文插座上。
- `_can_drop_data` / `_drop_data` 委托给上层 `EquipmentPanel.can_drop_rune_socket_data / drop_rune_socket_data`。
- 点击 → `select_skill_slot(slot)`，用于选中槽位预览。

### L5 持久化层
- `SkillRuntime.serialize()` 输出 `{slots, slot_runes, cooldowns}`；`deserialize()` 读回 `slot_runes` 并 `_ensure_slot_runes_shape()`（补齐/截断到 7 槽、按容量裁切）。
- 即符文镶嵌状态**已纳入存档**，但**全局存档系统（SaveManager）本身尚未实现**（见 `项目实现进度评估.md` §3 缺口）——目前 serialize/deserialize 已就绪，只差调用方。

### L6 获取层 — `roll_rune(source)`
- 按 `SOURCE_WEIGHTS[source]` 做轮盘抽奖，返回完整符文字典。
- 来源权重见 §4。

### 3.1 卢恩命名规范（设计决议：全部使用卢恩符文）

> **设计决议**：符文**全部使用卢恩文字（Elder Futhark / 古弗萨克）**作为标识。RPG 业界惯例（God of War、Hellblade、无数 roguelike）是用**单个有语义的弗萨克符**代表一种力量，而非把英文单词逐字母拼成卢恩。

**当前偏差**：§3 表中的 `卢恩文字` 列是**逐字母转写**（如 `ember`→ᛖᛗᛒᛖᚱ），不符合 RPG 惯例，也占 UI 空间。应改为**每个符文对应一个语义相符的单符**（见下表「建议映射」）。

## §5 修饰键参考（技能 ↔ 符文 接口契约）

符文通过 `mods` 键修饰技能定义里的数值字段。下列键被符文使用，其**基础值来源**见 `globals/combat/action_skills.gd`：

| 修饰键 | 含义 | 被哪些符文用 | 样例技能基础值 |
|---|---|---|---|
| `damage_mult` | 伤害倍率 | ember | 踢击 0.5 / 冲撞 0.8 / 抓取投掷 1.0 |
| `cooldown` | 冷却秒 | quick(-20%) / echo(+10%) | 踢击 2.0 / 冲撞 5.0 / 滑铲 3.0 |
| `cast_time` | 前摇秒 | quick | 冲撞 0.2 / 抓取投掷 0.4 / 战术滑步 0.1 |
| `knockback_m` | 击退距离(米) | force(×1.5) / launch(×1.35) | 踢击 3.0 / 冲撞 6.0 |
| `stun_sec` | 硬直秒 | force(+0.2) | 冲撞 0.5 / 抓取投掷 1.0 |
| `dash_speed_mps` | 冲撞位移速度(米/秒) | surge(+2.0) | 冲撞 12.0 |
| `physical_impact_damage_mult` | 地形撞击伤害倍率 | surge(+0.18) / launch(+0.20) | 踢击/冲撞 1.0（启用时） |
| `buff_value` | 增益数值 | guardian(×1.2) | 滑铲 iframes 0.5 / 战术滑步 dodge 0.3 |
| `buff_sec` | 增益持续秒 | guardian(+1.0) | 见各技能 buff_sec |

> **接口规则**：符文只认键名，不认技能。任何新技能只要在定义里提供上述键，就能被对应符文修饰——这是技能(15)与符文(19)之间的解耦契约。新增符文修饰键时，须同步在 §3 表与战斗层消费端登记。数值类被动的 `stat` 键（武器/盾面板层，见 `21` §2.3）是此契约的扩展集：二者**分层聚合、同一键不双写**（被动写基础面板 meta 层，符文写技能有效值 run 层）。

## §6 构筑协同与流派示例

符文**可同种叠加**至槽位容量上限（主动 3 / 被动 2），叠加按「先乘后加」结算。下列流派均已被 `rune_data_test.gd` 实测覆盖：

| 流派 | 槽位装配 | 效果（实测） | 定位 |
|---|---|---|---|
| **冲量位移流** | 冲撞 + surge×3 | `dash_speed_mps` 提升、`physical_impact_damage_mult` 提升（测试断言严格大于基础） | 撞墙/撞敌秒杀流 |
| **抛掷踢飞流** | 踢击 + launch×3 | `knockback_m` 提升、`physical_impact_damage_mult` 提升 | 把敌人踢飞撞墙流 |
| **爆发响应式** | 踢击 + ember+quick | `damage_mult` 0.5→0.6、`cooldown` 2.0→1.6、附加燃烧 | 高频爆发 |
| **控制链** | 冲撞 + force | `knockback_m ×1.5`、`stun_sec +0.2` | 击退+硬直控场 |
| **增益强化** | 被动槽 + guardian | `buff_value ×1.2`、`buff_sec +1.0` | 强化常驻被动/增益 |
| **额外触发** | 任意 + echo(rare) | `extra_projectiles:1` / `repeat_count:1`（机制标签，需战斗层消费） | 概率性倍率，代价 +10% 冷却 |

**协同要点**：
- surge / launch 都吃 `physical_impact_damage_mult` + 地形撞击(05) → 与「撞墙秒杀」机制强绑定，是构筑深度的高光区。
- ember 的 `burn_chance/burn_sec` 是**目前唯一的状态异常入口**（燃烧）——若要做元素流派，这是扩展锚点。
- echo 的 `extra_projectiles` / `repeat_count` 是**机制级**而非数值级，必须由战斗层显式消费，否则该符文「镶了没用」（见 §8 缺口 D5）。

## §11 技能-符文设计关系（设计规格）

> 本节为**设计规格**：重定义「被动技能 / 符文」的分工，并确立「符文局内获取」模型。
> 它与 §2/§6 既有「符文修饰主动技能数值」实现**并存**——详见 §11.4 的设计待决项。

### 11.1 三层职责重新划分

| 层 | 角色 | 回答的问题 |
|---|---|---|
| **被动技能（Passive Skill）** | **build 骨架**：分数值类 / 机制类（见 `21`） | 武器/角色「获得什么面板提升 / 什么操作被强化」 |
| **符文（Rune）** | **增强其效果** 或 **给与机制加强** | 面板/机制「多强 / 是否升阶」 |
| 主动技能 + 装备（既有） | 执行层 | 机制「如何触发」 |

- **被动技能 = 流派/build 定义层**，自 `21` 起**拆为两类**（二者都是 meta 持久，跨 run 保留）：
  - **数值类**：针对某武器/盾的纯乘加（`damage_mult`/`stagger_sec`/`block_value`…），**不是机制**，只改面板聚合点（见 `21` §2）。
  - **机制类**：选中一个已有操作（左键攻击/跳跃/格挡/投掷…）附加新行为（蓄力/二段跳/完美格挡增伤…），是"操作强化"（见 `21` §3）。
  - 旧稿"被动技能 = 给武器特殊机制"仅覆盖机制类，**已过期**——数值类被动不带任何机制，仅放大数值。
- **符文 = 放大/升阶层**：对已镶嵌被动（或其授予的机制）做 *增强（数值放大）* 或 *机制加强（升阶/附加态）*。它是 build 的肌肉。B 类符文放大被动机制（`21` §4）；A 类符文放大主动技能数值（`§6`）。
- 此分工让「同一把武器 + 不同被动」产生数值/操作差异，「同一被动 + 不同符文」产生强度/形态差异——构筑深度来自两层正交组合。

### 11.2 被动技能数据结构扩展（提议）

在 `SkillData` 被动定义中新增 `weapon_mechanic` 字段（当前被动仅为「武器流派被动 + 属性里程碑被动」，见 `skill_runtime.gd` 注释，无机制声明）：

```gdscript
# SkillData.SKILLS 中一个被动技能定义（提议结构）
{
    "id": "ember_touch", "name": "余烬之触", "type": "passive",
    "school": "ENCHANT_WAND", "tier": "T1",
    "weapon_mechanic": {                       # ← 新增：授予武器的特殊机制
        "id": "burn_on_hit",
        "type": "on_hit_status",                # 机制类型（消费端按 type 分发）
        "params": {"chance": 0.20, "sec": 2.0, "dps": 3.0}
    }
}
```

- **消费端**：装备/战斗层读取当前 5 个被动槽集合，聚合所有 `weapon_mechanic`，应用到**手持武器**（而非技能实体）。一处机制可被多被动叠加。
- 与现有「双轨领悟阶梯」（school + tier，按熟练度/属性解锁，见 `skill_data.gd`）天然契合：每个流派/阶位对应一种机制授予。
- **机制注册表（骨架层）**：**武器层 × 流派层 双层模型**（武器固有机制 + 流派机制，可跨武器共享如持盾流派）的完整 `weapon_mechanic` 定义（id / type / params / 各机制彼此不同）见 **`20-被动机制树（构筑骨架）.md`**。本文（19）是「乘数层」，20 是「骨架层」，二者互链。
- **被动技能分类（数值类 / 机制类）**：机制类被动（蓄力 / 空中二段跳 / 撞敌 / 冷却缩短…操作强化）与数值类被动（武器/盾数值提升）的设计见 **`21-被动技能（数值类与机制类）.md`**；B 类符文对其放大/升级（§4）。

### 11.3 符文对被动机制的两种作用（由 **B 类 · 被动机制符文** 执行）

> 见 §11.4：符文分两类。对「被动授予的武器机制」做放大/加强的，是**独立的 B 类符文池**，与既有的 A 类主动数值符文（§6）不是同一种东西——不要把现有 ember/echo 等当成被动机制符文。

- **(A) 增强效果（Amplify）**：对 `weapon_mechanic.params` 做标量修饰——复用 `mods` 的 `{"mul":x}` / `{"add":y}` 体系。
  - 例（B 类符文「灼燃」）：`burn_on_hit.params.dps ×1.2` 且 `chance +0.25`。
- **(B) 机制加强（Upgrade）**：升阶或附加次级机制——需在 `rune_data.gd` 新增 `mechanic_upgrade` 字段（区别于现有纯数值 `mechanics` 标签）。
  - 例（B 类符文「蔓延」）：`burn_on_hit` 升级为「燃烧蔓延至相邻敌人」或解锁第二机制 `ignite_on_kill`。
  - 机制加强是**定性**变化，必须由消费端显式支持对应的升级键（否则符文「镶了没用」，同 D5 风险）。

### 11.4 两类符文（设计决议 D10 已定：共存，是两类不同符文）

符文**不是**「一个池子两种用法」，而是**两个独立的符文类别**，各自对应不同槽位与不同修饰目标：

| 类别 | 镶嵌槽位 | 修饰目标 | 作用方式 | 现状 |
|---|---|---|---|---|
| **A 类 · 主动数值符文** | 主动技能槽（F/G，容量 3） | 主动技能数值 | 数值乘/加（§6：surge×3 冲撞、launch×3 踢击、ember 增伤、quick 降冷却） | 现有 7 符文（ember/quick/force/surge/launch/echo/guardian）主体即此类的雏形 |
| **B 类 · 被动机制符文** | 被动技能槽（5 槽，容量 2） | 被动授予的武器机制 | 增强(参数缩放) / 机制加强(`mechanic_upgrade`)（§11.3） | **待定义**：独立符文池，需补名称 / mods / mechanic_upgrade 规格 |

- **共存**：两类符文同时存在、各自成池；run 内从对应池获取，镶嵌到对应槽位。
- **槽位绑定**：主动槽只接受 A 类，被动槽只接受 B 类（`socket_rune` 按 `rune_class` 字段分流校验）。
- **数据层**：`rune_data.gd` 每项符文新增 `rune_class` 字段（`"active"` / `"passive"`）；`roll_rune` 可按类别分别抽取或加权。

> **D10 已决议**：不二选一 —— 双通道**共存**，且明确为「**两类不同符文**」而非「同一符文两种用法」。既有的主动数值修饰（§6）完整保留，被动机制加强（§11.3）作为第二类独立存在。A 类与 B 类**不可混镶**到对方槽位。

### 11.5 装配矩阵（新）

```
【A 类 · 主动数值符文 → 主动技能】
冲撞 + surge×3        → dash_speed + 冲撞撞击伤害（§6 实测）
踢击 + launch×3       → knockback + 落点撞击伤害（§6 实测）

【B 类 · 被动机制符文 → 被动授予的武器机制】
余烬之触(burn_on_hit) + 灼燃(增强)     → 燃烧 dps×1.2 + 25% 概率
余烬之触(burn_on_hit) + 蔓延(加强)     → 燃烧蔓延 / 击杀点燃
迅捷步法(dodge_iframe)+ 守护(增强)     → 无敌帧 +0.2s / 持续 +1s
```
> 注：上表「灼燃 / 蔓延 / 守护(被动向)」为 **B 类待定义符文**示例名，最终名称与规格见 G9。

## §12 局内获取模型（roguelike run-scoped）

### 12.1 设计意图
**符文为局内获取**：符文是**一次地牢探索 run 内**获取的临时强化，不是元进度仓库物资。这与逃脱符石（04，局内撤离道具）的「局内」属性一致，但二者生命周期不同（见 §0 / D8）。

### 12.2 与现有代码的冲突（需改造）
当前 `tavern_equipment_panel.gd` 存在 **持久化仓库视角**的符文库存：
- `_get_carried_rune_inventory()`（随身）+ `_get_warehouse_rune_inventory()`（仓库）+ `_combined_owned_rune_inventory()`（持有合计）。

这与「局内获取」冲突，且 §12.5 已决议**回酒馆即全部丢弃**——故需引入 **run-scoped** 状态并**彻底废除持久化仓库视角**：

- 新增 `RunRuneState`：本次 run 内「已获取符文池」+「已镶嵌状态」，是唯一持有方。
- **run 开始**（进入地牢）→ 清空 rune 池与镶嵌（`SkillRuntime.reset()` 已重置 `slot_runes`，需确保它挂在 run 生命周期而非仅存档重置）。
- **run 中** → 宝箱/精英/Boss 掉落调用 `RuneData.roll_rune(source)` 入 run 池。
- **run 结束**（撤离成功或阵亡，回到酒馆）→ `RunRuneState` 与镶嵌**全部丢弃**，零保留。
- **废弃项**：`tavern_equipment_panel.gd` 的 `carried_rune_inventory` / `warehouse_rune_inventory` / `_combined_owned_rune_inventory()` 整组删除——符文不再有跨 run 库存。

> 现状：`procedural_dungeon.gd` 无显式 run-reset 钩子（grep 未见 `start_run/end_run`），需新增 run 生命周期事件供 `RunRuneState` 订阅。

### 12.3 局内获取触点
| 触点 | 来源 (roll_rune source) | 备注 |
|---|---|---|
| 宝箱开启 | chest | 偏向 common（ember/quick） |
| 精英击杀掉落 | elite | 偏向 combat 向（force/surge/launch） |
| Boss 击杀掉落 | boss | 高概率 rare（echo） |
| 符文祭坛 / 事件 | event（待加） | 设计可选触点 |

### 12.4 镶嵌时机与 UI
- 局内随时可在装备面板/快捷菜单 socket（已有 `RuneSlotDropButton` 拖拽插座，需从「镇上仓库视角」切到「局内 run 视角」）。
- 镶嵌 UI 数据源由 `_combined_owned_rune_inventory()` 改为 `RunRuneState` 的 run 池。

### 12.5 元进度策略 — **已决议：回到酒馆即全部丢弃**

> **设计决议（D11 已定）**：符文是局内临时强化，**无论撤离成功还是阵亡，只要回到酒馆，本次 run 获取的全部符文一律丢弃，零 meta 保留（纯 roguelike）**。

- 符文与「meta 成长资产」（酒馆升级、声望等，见 `08-酒馆成长系统.md`）彻底解耦：**符文不进仓库、不跨 run 携带**。
- 对 §12.2 的影响：`tavern_equipment_panel.gd` 的 `carried_rune_inventory` / `warehouse_rune_inventory` / `_combined_owned_rune_inventory()` **整组废弃**，符文不再有持久化库存概念，`RunRuneState` 是唯一持有方，随回酒馆事件清空。
- 设计含义：每次下地牢都是「从零构筑」的 roguelike 体验；玩家的 meta 成长只来自酒馆经营/声望（08），符文是单次 run 内的强度表达。
- **与此对照**：被动技能本身（来自双轨领悟阶梯，按熟练度/属性解锁，见 `15-技能与领悟系统.md`）属于 **meta 持久进度**，不随 run 丢弃；被丢弃的仅是「局内获取的符文」及其镶嵌。即「被动机制框架常驻、符文临时放大」的清晰分层。
