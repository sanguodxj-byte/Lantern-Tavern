## 一、首轮遗漏修正：火把音频暂停其实没生效

首轮在 `dungeon_streaming_controller.gd` 加了 `_apply_visual_side_effects`（隐藏时暂停粒子/音频），但**只对「注册为可视节点」的对象生效**。

核查发现：`dungeon_scene_builder.gd:_spawn_torch_on_wall`（:822）把火把加进了 `result.streamed_physics_nodes`（即**只注册为物理节点**，没有调 `register_streamed_visual_node`）。因此火把的 `AudioStreamPlayer3D(autoplay=true)` 从开局起**永不暂停**——远距火把仍常播火焰音。首轮修复对火把无效。

**本轮补漏**：在 `_set_visual_root_active`（物理激活路径）对根节点也调用 `_apply_visual_side_effects(root, not active)`，使物理注册节点（火把等）在离开激活半径时同样暂停其下粒子与音频。已落地。

## 二、本轮新发现（按严重度）

### 🔴 P-A 地牢生成仍是单帧卡顿根因（首轮只修了敌人，没修生成）
- `procedural_dungeon._ready()` 同步串行执行：`generate → 校验 → build()`（地形/碰撞/装饰/火把 prefab 实例化/导航烘焙）→ `runtime.start()`。
- `dungeon_scene_builder._build_navigation_mesh`（:720）`NavigationServer3D.bake_from_source_geometry_data(..., Callable())` 是**空回调 = 阻塞式同步烤整图导航网格**（最多 2048 地板面），是进场最长单帧 stall 的主要来源。
- 首轮把"敌人实例化"分帧了，但**几何 + 装饰 + 导航烘焙仍在加载帧一次性同步完成**。
- 方向：把 `build()` 拆为"先骨架后重资产"并分帧；导航烘焙改异步（传非空 `Callable` 后台烤），首帧用占位/延迟激活敌人寻路。

### 🔴 P-B 单全局 navmesh + 多敌同步 `set_target_position` → A* 雷群周期尖峰
- `enemy_state_moving.gd`：每物理帧 `get_next_path_position()`（路径跟随，廉价），但 `set_target_position` 触发全图 A*，每 150ms（`PATH_UPDATE_INTERVAL_MS`）一次。
- 所有敌人 `last_path_update_time` 初值相同（-150），同批生成计时对齐 → **同一帧集体重算路径** → 20–40 敌同帧对整图 navmesh 做 A* 的周期尖峰。
- 方向：`set_target_position` 加随机抖动（0–150ms 错开）；仅在玩家位移超阈值或到达终点时重设目标；远距 imposter 敌人降频/停用导航。

### 🟠 P-C 物理/AI 激活半径 ~36m（9 chunk），与 imposter LOD 18m 严重失衡
- `STREAM_CHUNK_SIZE_CELLS=8 × tile_size=3 = 24m/chunk`；`STREAM_PHYSICS_CHUNK_RADIUS=1` → `_iter_chunks` 覆盖 3×3=9 chunk ≈ **72m×72m（玩家周围 ~36m 半径）**。
- imposter LOD 在 **18m** 即隐藏蒙皮网格换 billboard 并暂停动画，但 18–36m 这圈敌人**仍在跑完整 AI/寻路/物理**——CPU 在算玩家看不见的东西。
- 方向：把"AI 模拟半径"从"物理激活半径"解耦（如 16–20m 或更细 chunk）；对 imposter 带且未追击的敌人降频 `_physics_process` 或停 `NavigationAgent3D`。

### 🟠 P-D 火把/非批装饰为独立 MeshInstance3D，未 MultiMesh 合批 → draw call 高
- 火把（`_spawn_torch_on_wall` `TORCH_PREFAB.instantiate()`）每把是完整场景实例，含 1 个 `MeshInstance3D` + 1 个 `GPUParticles3D`；火把**不在** `batched_decor_scenes` 列表，不走 MultiMesh。
- 大地图可能 30–60 把独立火把 → 高 draw call + 高粒子系统数（已修音频，但绘制/粒子系统数仍可合批优化）。
- 方向：把火把纳入 `batched_decor_scenes` 走 MultiMesh；或给"活跃火焰"也设预算（类似灯光预算 12）。

### 🟠 P-E Shader 首次编译卡顿（无预热）
- 地形 ShaderMaterial 多套 tile 变体、各敌人 GLB 内嵌材质、blood/metal_spark/voxel_chip 各有专属 ShaderMaterial。
- 敌人分帧生成（每帧 4 个）使每个新敌种 GLB 材质首次出现即触发一次编译 hitch，叠加在分帧生成上。
- 方向：启用 Godot shader 预编译/变体缓存；加载遮罩后预实例化首批敌人做材质 warming；合并材质变体。

### 🟡 P-F 追击敌每物理帧 `weapon_reach_raycast.is_colliding()` 射线
- `enemy.is_player_within_reach()`（:416）每个近战追击敌每物理帧查一次 `RayCast3D`。单条廉价，但叠加在导航/LOS 之上；大量近战贴脸时可见。
- 方向：距离平方粗筛 + 仅攻击前摇做精确射线；或同帧多敌设射线预算上限。

### 🟡 P-G `fx_helper` 每次命中 instantiate 多个特效 prefab（瞬时尖峰）
- 每次受击：blood_spurt + voxel_chip + damage_number 三个 prefab 各 `instantiate()`。各特效已自销毁（非泄漏），但重战（20–40 敌混战、每秒数十次命中）有瞬时 instantiate 尖峰。
- 方向：blood/damage_number 这类高频短寿命节点做**对象池**复用。

## 三、澄清：两项"假设热点"其实不是

- **联机 `tick()` 每帧成本**：单人模式 `session == null` → `tick()` 直接 early-return，几乎零成本；`_run_detection` 在代码库中**不存在**（仅旧文档提及）。多人 host 下 peer 上限 4，`_flush_snapshots` 已 30Hz 节流。非热点。
- **`exploration_pressure` 每帧成本**：每帧只累加计时器，仅跨过整分钟才推进。非热点。

## 五、实施状态（2026-07-21 第三轮 · 继续优化）

在「首轮 5 项 + 第二轮补漏（火把音频暂停）」之上，本轮落地 P-B / P-A / P-C 三项。

### ✅ P-B navmesh 雷群抖动（已生效）
- 文件：`scenes/characters/enemies/state/enemy_state_moving.gd`
- 改动：
  1. 新增常量 `PATH_UPDATE_JITTER_MS := 50` 与每实例 `var _path_update_interval_ms := PATH_UPDATE_INTERVAL_MS + randi_range(0, PATH_UPDATE_JITTER_MS)`。
  2. 初始相位随机化（`last_path_update_time := -randi_range(0, PATH_UPDATE_INTERVAL_MS)`），使同批敌人首次重算分散在不同帧。
  3. 节流比较由 `>= PATH_UPDATE_INTERVAL_MS` 改为 `>= _path_update_interval_ms`（追击 + 巡逻两处），间隔本身各异 → 长期持续漂移错开，杜绝同帧集体 `set_target_position`（整图 A*）周期尖峰。
- 验证：headless 解析校验通过。

### 🟡 P-A 生成/导航异步烘焙（代码就绪，默认关闭待窗口化验证）
- 文件：`scenes/expedition/dungeon_scene_builder.gd` → `_build_navigation_mesh`
- 改动：新增常量 `ENABLE_ASYNC_NAVMESH_BAKE := false`（默认关闭）。开启时 `NavigationServer3D.bake_from_source_geometry_data_async(...)` 后台线程烘焙，消除进场最长单帧 stall；缺 API 时回退同步分支。异步方法名含 `bake_from_source_geometry_data` 前缀，`perf_optimization_test.gd` 仍通过。
- 为什么默认关闭：本环境仅 headless，导航烘焙文档不稳定（偶发 native crash；`--script` 下异步完成回调不触发，验证探针挂起无法判定回填正确性）。异步若未能把多边形正确回填进 `NavigationRegion3D`，敌人将永久无法寻路（灾难性且不可在此环境验证）。故保守默认同步（当前生产已知可用），待「有窗口」构建冒烟测试通过后再置 `true`。
- 验证：解析校验通过；窗口化冒烟测试为发布前置条件。

### ✅ P-C AI 半径与 LOD 对齐（已生效）
- 文件：`scenes/characters/enemies/enemy.gd` + `enemy_state_moving.gd`
- 改动：
  1. `enemy.gd` 新增常量 `AI_SIM_RADIUS_M := 18.0`（与 `ENEMY_IMPOSTER_LOD_DISTANCE` 18m 边界对齐）与方法 `is_ai_active()`：已与玩家交战 / 暗蚀强制追击 / 玩家进入 18m 内 → true；远距未交战替身带 → false。
  2. `enemy_state_moving._physics_process` 开头加门：`is_ai_active()` 为 false 时仅清零水平速度并 `process_movement(delta)` 保持物理静止，跳过追击/巡逻的 A* 与导航查询。
- 效果：把「AI 模拟半径」从 ~36m 物理激活半径解耦到 18m LOD 边界，削减 18–36m 替身带敌人空算寻路/动画的 CPU 浪费；敌人本体物理碰撞仍保留。
- 验证：headless 解析校验通过。边界互补（LOD 在 >18m 开启，AI 在 >18m 停用），玩家逼近 18m 时敌人恰好重新激活，无可见冻结。
