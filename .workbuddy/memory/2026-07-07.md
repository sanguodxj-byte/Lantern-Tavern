## UI bug 修复第三轮：全屏透明 STOP 控件拦截点击（真正根因，已完成）
- 现象：前两轮分别修了 CombatHUD 泄露和 player.gd 鼠标强制捕获，但经营界面仍"完全无法操作"——鼠标光标可见但所有点击都不响应。
- **真正根因**：UI CanvasLayer(layer=20) 内的 DeathScreen(ColorRect) 和 HurtVignette(Panel) 是**全屏透明控件**（modulate alpha=0 但 visible=true），mouse_filter 默认 STOP。它们在 layer=20 上，高于 tavern_hud_layer(原 16)，**拦截了所有鼠标点击**——鼠标光标可见但每次点击都被透明全屏控件吞掉。
- 修法：
  - `ui.tscn`：DeathScreen 和 HurtVignette 加 `mouse_filter=2(IGNORE)`——它们是纯视觉覆盖层，不应拦截输入。
  - `tavern_manager_node.gd`：tavern_hud_layer.layer 从 16 提到 25（高于 UI 的 20，防御性措施）。
- 测试：`tavern_hud_mouse_bug_test.gd` 加 2 例运行时契约（实例化 ui.tscn 验证 DeathScreen/HurtVignette 的 mouse_filter==IGNORE）。全量 5 套件 67 例 0 失败。
- 教训：**"UI 可见但无法点击"优先排查全屏透明 STOP 控件**，而非鼠标捕获。Godot 中 ColorRect/Panel 默认 mouse_filter=STOP，即使 modulate alpha=0 也会拦截输入。

## 找到"模型方向不正确"根因并修复（已完成）
用户定位：**skeleton，模型方向不正确**。之前的 8 项 bone/skeleton 朝向测试都 UPRIGHT 是因为**只查了骨骼没查网格**——骨骼 Y-up 正确，但**网格几何 Z-up（躺）**。
- **根因**：源 `voxel_*.glb` 的网格几何是 Z-up（躺），骨骼是 Y-up（立）——源资产本身网格/骨骼空间不一致（这批 voxel GLB 从未真接入场景，没人发现）。我的 rig 流程把肢体方块 join 成单蒙皮网格后，顶点保持 Z-up（躺），bind 到 Y-up 骨 → 渲染时网格躺、骨骼立 → "躺地上摆动手脚"（run 动画驱动立骨，蒙皮网格躺，肢体被拉扯）。

## 体素模型朝向修正：正面从 +Z 翻转到 -Z（已完成）
用户反馈"各个实体移动时朝向都不正确"。排查发现**不是"躺着"而是"背对目标"**。
- **根因**：体素模型在 Blender 中正面朝 -Y（眼睛/鼻子在 -Y 侧），导出 `export_yup=True` 后 -Y → +Z（glTF/Godot **后方**）。但 Godot 的 `looking_at(target)` 让 -Z 指向目标，模型正面在 +Z → **背对目标**。鼠/龙正面在 -X（Blender）→ -X（Godot 左侧），同样不对。
- **关键区分**：与上一条"网格 Z-up 躺着"不同——那个是 OLD `rig_creature.py` 的蒙皮网格空间不一致问题，已在旧脚本中修复。本次是 NEW `generate_voxel_creature_rigs.py` 的**朝向方向**问题（模型立着但面朝错误方向）。
- **修复**：`tools/generate_voxel_creature_rigs.py` 在导出前给根节点添加 Z 轴旋转（不 apply，保留为对象级旋转，骨骼局部空间不变，动画不受影响）：
  - 人形 + 史莱姆: `root.rotation_euler = (0, 0, math.pi)` → 180° Z (Blender) → 180° Y (glTF)，+Z 翻转到 -Z ✓
  - 鼠 + 龙: `root.rotation_euler = (0, 0, -math.pi/2)` → -90° Z (Blender) → -90° Y (glTF)，-X 旋转到 -Z ✓
- **重生成**：Blender 管线重跑 all，7 个 `_rig.glb` 全部导出成功，日志含 `facing correction: rotated root by -90.0° Z` / `180.0° Z`。
- **测试**：`voxel_rig_animation_test.gd` 新增 2 例：`test_humanoid_rig_glbs_have_facing_rotation`（递归搜索节点树验证 180° Y）、`test_creature_rig_glbs_have_facing_rotation`（slime 180° / rat+dragon -90°）。全 20 例 0 失败。
- **注意**：180° Y 旋转会镜像动画（右手挥砍变左手挥砍）和武器位置（Hand.R 骨骼视觉上在左侧），但 gameplay 不受影响（hitbox 独立于视觉模型）。如需彻底修复需在 Blender 源脚本中改模型正面方向（-Y → +Y），但改动量大。
