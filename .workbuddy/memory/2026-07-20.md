# 2026-07-20

## 玩家动画架构梳理 + 左键漏出"角色一部分"根因
- 玩家动画为两层解耦：第三人称身体(character = voxel_player_48px_rig.glb, 15骨/18动画, 无AnimationTree, 由 player_state_* 直接 play(name)) 在第10渲染层被主摄像机隐藏；第一人称 ViewModel(武器GLB, 无手臂) 用独立 SubViewport 武器摄像机(第11层) 经 CanvasLayer 合成。战斗状态机权威，ViewModel 仅视觉采样。
- 多动作模组：身体不支持多rig(玩家+9敌人共用同一 character.glb，仅配色/缩放差异；7个体素敌人GLB为悬空引用未接入)。FP ViewModel 支持按武器类型多动作集(vm_slash_one_hand/heavy/stab_dagger/thrust_spear/bow/crossbow/wand)。
- 左键看见"角色一部分" = 手臂模型（用户纠正：不是刀光）。但干净构建下身体隐藏正确：headless 层扫描(player 内嵌主场景,autoload就绪) 显示身体全体在第10层；窗口化 fp_idle/fp_slash 截图(此前成功跑出) 只见 FP 武器、无任何身体/手臂。故当前源码不会复现该 bug。
- 曾误判根因为 SlashTrail 刀光漏出(layer1)，并给 combat_slash_animator 加了 trail_layer 参数把玩家刀光推到第10层——结果把玩家自己的刀光藏没了(用户报"刀光根本不可见")。该错误修复已撤销，玩家刀光恢复第1层可见。
- 真因方向：`player.gd _ready` 原顺序为 ANIM_REPAIR.repair_all(183行) → _hide_character_body(186行)。隐藏身体排在可能抛错的初始化之后；若 _ready 在 186 前中断(PhysicsSetup/repair_all/装备异常)，身体停留第1层→主摄像机可见→左键挥砍手臂入镜。属脆弱的顺序依赖。
- 防御性修复(已落实)：把 `_hide_character_body()` 提到 `_ready` 最前(equipment_preview 提前 return 之后、PhysicsSetup 之前)。headless 复跑 probe_leak_scan 确认身体仍正确隐藏(无 LEAK@layer1)。窗口化截图因沙箱拦截 NVIDIA/D3D 缓存无法再跑，但此前 fp_idle/fp_slash 已证干净场景无手臂。
- 架构清晰度评估(用户问"架构是否不清晰")：顶层 FP 两层解耦模式清楚合理；但"第一人称下谁对主相机可见"的约束是散的、靠巧合维持，最可能是真因：① 身体不可见是 _ready 调用顺序的副作用而非强制契约(已用提前隐藏缓解，但治标)；② 渲染层约定(1世界/10身体/11FP武器)散落多文件，且 ViewModel 武器默认 layer1 仅相机同步后搬11、刀光 MeshInstance3D 创建不设 layer→默认1，无中央强制"FP几何绝不在layer1"；③ 身体+ViewModel 两套平行模型可见性被复制三处(身体10/武器11/刀光1)无单一归属。建议治本：抽 FirstPersonVisibility 管理器或在 _ready 开头+断言确认身体在第10层，所有 FP 几何统一走 set_first_person_only(node) 助手。待用户确认修复后是否做架构级收口。
- **【已实机验证】用户回"目前正常的"** → 把 `_hide_character_body()` 提到 `_ready` 最前的修复在真实游戏中生效，"左键露手臂"消失。据此根因实锤：`_ready` 原顺序(ANIM_REPAIR.repair_all 在隐藏身体之前)使隐藏身体排在可能抛错的初始化之后，真实运行某前置步骤异常中断导致身体停留第1层。顺序依赖假设成立。治本(集中 FP 可见性管理器)仍可选，但症状已消除。
