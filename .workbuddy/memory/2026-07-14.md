## Phase 5 完成：存档信任边界 / SaveAuthority 幂等（防重复 extract/save 刷物品）
- 根因（§17.4 安全测试"重复拾取/修改库存"收口）：原 `_handle_extract` / `_handle_save` 无结算序列校验，重复发命令会重复把玩家【本次净获得】写回单人存档 → 刷物品漏洞。桥接层每次都累加净获得。
- 修复：`globals/multiplayer/save_authority.gd` 由占位重写为 `extends RefCounted` 幂等账本（无 class_name，preload 访问）。以 **player_guid**（稳定身份，不随 peer_id 变）为主键维护「是否结算 + settlement 快照」：
  - `begin_expedition(revision=-1)`：revision>=0 用传入值（对齐 `DungeonAuthority.layout_revision`）否则自增；清空 `_settled`。
  - `is_settled(guid)`：空 guid→false；记录存在且 revision 匹配→true。
  - `mark_settled(guid, settlement)`：已结算返回 false（幂等 no-op）；否则深拷贝记录返回 true。
  - `get_settlement` 返回深拷贝（防外部污染）；`migrate_guid` 为 no-op（账本以 guid 为键，peer_id 变化不影响）；`serialize/deserialize` 含 expedition_revision + settled。
- `session_root.gd` 接线：`_init()` 建 `save_auth`；`CMD_EXTRACT`/`CMD_SAVE` 统一走 `_settle_expedition(peer_id, ctx, requested_save)`：用 `_peer_guid()` 取 guid，已结算则返回缓存 settlement + `already_settled=true`，否则计算净获得、`mark_settled`、返回 `already_settled=false`。`_handle_start_expedition` 调 `save_auth.begin_expedition(dungeon_auth.layout_revision)`。`build_session_snapshot` 含 `save=` 段、`apply_session_snapshot` 还原 → 防「断线→重连→再结算」绕过幂等。
- `multiplayer_scene_bridge.gd`：`EVT_EXTRACTION_RESULT` 处理加 `not bool(event.get("already_settled", false))` 守卫，重复结算时跳过 `_apply_extraction_settlement` 写回。

## 多进程集成测试结论：沙箱 loopback ENet 抖动，非 Phase 5 回归
- 直接 `run_in_background=true` 的 PowerShell 任务在 ~2min 被强杀（双/三进程 ENet 测试需 5~12min）。改走 **PowerShell 工具前台调用**（自动转后台持续跑完）才拿到完整输出。
- `run_dedicated_server_test.ps1` 完整跑完（exit=0）：`dedicated server ready: OK port=30021`、`client launched`、但 `CLIENT: FAIL connect (no dedicated server on port 30021)`。server.log 确证 `listening on port 30021 (max 8 players)`，客户端 `mp_dedicated_server_test.gd:99 NetworkManager.join("127.0.0.1", try_port)` 用标准 loopback。→ 服务端监听正常、客户端连不上，属**本沙箱跨进程 ENet loopback 偶发不通**（此前同代码 `DEDICATED SERVER TEST: PASS` 过；Phase 5 未动任何 networking 代码）。
- `run_integration_dungeon_test.ps1` 同样现象：host 进程起来且 NETSTATS 见 `player_despawned=1`（有 peer 连上即掉），但 client1/client2 未写出 client_ready/client_ok 信号文件。

- 已清理残留 Godot 进程与 `.tmp_itest_*`/`.tmp_apdata_*` 临时目录；运维铁律（2min 后台上限 / loopback 抖动判定）已补进 MEMORY.md。

## Phase 7 完成：客户端心跳调度（补全断线判定闭环的最后一块拼图）
- 根因：服务器侧断线判定基础设施早已齐备——`NetworkManager.send_heartbeat()`(unreliable RPC)→`_server_handle_heartbeat`→`SessionRoot.heartbeat`→`ConnectionAuthority.touch`(刷新 `last_seen`)；`NetworkManager.tick(delta)`（`procedural_dungeon.gd`/`dedicated_server.gd` 每帧调）推进 `current_time` 并对在线 peer 跑 `check_timeout`(`now-last_seen > HEARTBEAT_TIMEOUT=15s`→走 `handle_peer_disconnected` 进 GRACE 保留期）。**但全仓零调用 `send_heartbeat()`**——客户端从不 ping，理论上远端客户端若真卡死/掉线，服务器永远靠 ENet 断线事件兜底，心跳机制形同虚设。
- 修复：`scenes/multiplayer/client_command_driver.gd` 新增 delta 累加器调度：
  - `const HEARTBEAT_INTERVAL := 5.0`（远低于 15s 超时，保证超时前 2~3 次 ping）。
  - `_physics_process` 顶部（独立于 player 节点状态）每帧调 `_maybe_send_heartbeat(delta)`。
  - `_maybe_send_heartbeat`：累加 delta，达间隔则 `nm.send_heartbeat()` 并归零；房主(is_host，自身即服务器)不向自己 ping，仅远端客户端(is_client)触发；单帧超大 delta 只发 1 次（不补发，防瞬间风暴）；nm 为 null/未激活则跳过（无真实 NetworkManager 时安全 no-op）。
- **测试坑（重要，已记入 MEMORY.md）**：Godot 4.7 **运行时强制校验带类型注解的函数参数**！`_maybe_send_heartbeat(delta, nm_override)` 参数原标 `Node`，但测试 mock `MockNM extends RefCounted`→报 `Invalid type... not a subclass of the expected argument class`。解决：参数+局部变量都改 `Object`（真实 NetworkManager 是 Node⊂Object、mock 是 RefCounted⊂Object 都满足）。**凡测试需注入 mock 的函数参数，一律用 `Object`/`Variant`，勿用 `Node`**。
- **验证（gdUnit 全绿，零回归）**：
  - 新建 `tests/gdunit/client_command_driver_test.gd`：**7 cases | 0 errors | 0 failures | PASSED 101ms**（间隔前不触发 / 跨间隔触发1次 / 累加器归零 / 非客户端跳过 / 未激活跳过 / 超大delta只1次 / 端到端心跳刷新服务器 last_seen）。
  - 回归：`connection_authority_test`(29) / `network_manager_integration_test`(9,exit101) / `session_root_test`(59,exit101) 全部 0 errors 0 failures。
- MEMORY.md 新增「客户端心跳调度(Phase 7)」铁律条目 + GDScript 4.7 运行时强校验带类型参数的 gotcha（测试注入 mock 用 Object/Variant）。
