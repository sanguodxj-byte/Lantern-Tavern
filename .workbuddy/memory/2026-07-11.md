## 联机可行性评估（用户问「当前架构对联机支持如何」）
- 结论: **当前架构=深度单机, 联机需大改, 几乎重写核心状态/战斗/输入层**。游戏代码零联机(仅 gdUnit4 addon 自带 TCP 与游戏无关)。
- 根因: ① `GameState.current_player` 全局单一玩家引用; equipment_loadout/expedition_inventory 全局单例 → 假设恰好一个本地玩家。② 28 个 autoload 单例(AttrPanel 存"那一个"玩家属性/CombatEngine 直接改血/SkillRuntime 全局技能状态) → 无 per-peer 实例概念。③ 战斗/伤害直接函数调用改状态, 无 authority/RPC/复制层。④ 输入 player._input 直接驱动本地唯一 Player, 无 multiplayer_authority 门控。
- 若要联机: 需引入 per-peer 玩家状态容器(拆 GameState/AttrPanel 单例)、MultiplayerSynchronizer/Spawner 复制敌人+掉落+门/宝箱、伤害走服务器权威+RPC、输入按 authority 门控、存档改多人。属"架构级重写", 非增量。
- 现状定位: 单机优先合理(Barony 虽原生 co-op 但本作未走此路); 若未来要 co-op 建议尽早在 GameState/AttrPanel 抽象"玩家上下文", 越晚改成本越高。
