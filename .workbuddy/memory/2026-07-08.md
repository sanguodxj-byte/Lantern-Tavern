## P4 火把实时阴影修复「隔墙漏光」（用户选定方案：预算内火把开实时阴影）
- 现象：P0 遮挡剔除把墙后火把网格/火焰剔除了，但其点光源球状影响范围无视墙体几何，仍照亮墙另一侧 → 用户报「大量光源隔墙渲染/透到墙另一面」。
- 根因：地牢/酒馆点光源此前 `shadow_enabled=false`，OmniLight3D 以 11m 半径球投光、不读深度。
- 修复 `voxel_prop.gd` `_build_torch`：火把光 `shadow_enabled=true` + `omni_shadow_mode=1`（**cubemap**，每灯 6 次深度绘制；0=双抛物面 2 次）+ `shadow_bias=0.03`/`shadow_normal_bias=0.3`。
- 预算内范围：开销由 `_update_streamed_lights` 收敛——仅最近 DUNGEON_VISIBLE_LOCAL_LIGHT_BUDGET(12) 盏 `light.visible=true`，不可见火把不渲染阴影，cubemap 阴影代价只落预算内火把。酒馆火把始终可见(常驻阴影)。
- **Godot 4 API 踩坑（关键）**：本仓 Godot 4.7 的 OmniLight3D **没有** `shadow_atlas_resolution` 属性（Godot 4 已把点光阴影分辨率改为渲染器全局阴影图集控制，per-light 属性移除）；且枚举常量 `OMNI_SHADOW_DUAL_PARABOLOID` / `OMNI_SHADOW_CUBEMAP` 也不以该名暴露。正确做法：直接赋值整数 `omni_shadow_mode = 0`（双抛物面）或 `= 1`（cubemap）。有效属性经 `get_property_list()` 探针确认：`shadow_enabled/shadow_bias/shadow_normal_bias/shadow_reverse_cull_face/shadow_transmittance_bias/shadow_opacity/shadow_blur/shadow_caster_mask/distance_fade_shadow/omni_shadow_mode`。
- 探针脚本写法坑：`-s` 的 SceneTree 脚本在 `_init()` 里 `quit()` 会挂起不退出；须用 `func _initialize()` 入口 + `quit()`。且 `tail -N` 会吞掉中间 `print` 输出，应重定向到文件再 Grep。
- 测试 torch_optimization_test.gd 现 5/5（新增 test_torch_light_casts_real_time_shadow 实例化 torch.tscn 断言 `shadow_enabled==true` & `omni_shadow_mode==0`；源码断言 `shadow_enabled = true` 与 `omni_shadow_mode = 0`）。

## Minecraft 骨骼/动画架构复核（用户问"几百个性能都没问题"）
- 关键澄清：Minecraft 的"骨骼"**不是骨骼蒙皮(skinning)**。官方文档（MicrosoftDocs/minecraft-creator `EntityModelingAndAnimation.md`）："Bones are the part of the model that can be animated, and they can be arranged in an arbitrarily deep hierarchy. Each bone itself is invisible but can contain cubes"。即骨骼=不可见变换层级节点，每个骨骼挂**刚性立方体(cubes)**；动画=平移/旋转/缩放节点，立方体随之刚体移动，**无逐顶点蒙皮变形**。
- 为什么几百个也流畅：① 无蒙皮数学（最贵部分被去掉），每实体≈10–30 盒，动画=每帧≈10–30 次矩阵更新；② 低面数(盒)；③ 视锥剔除 + 实体通常无实时阴影；④ 现代 GPU 画几千盒 trivial；Bedrock 还有 render controller 批量。
- 三范式对照：**Barony 预烘焙帧交换**（无蒙皮、CPU≈0，但动作离散）／**Minecraft 刚性骨骼盒**（无蒙皮、CPU≈0、动作平滑，最适合方块/体素角色）／**我们 Skeleton3D+SkinnedMesh3D 加权顶点蒙皮**（连续网格、每帧蒙皮、成本∝顶点数×骨骼数，近敌是瓶颈）。
- **新增洞察（AI 出图即出角色）**：MC 式 AI 皮肤工作流可行，根因是「所有角色共享固定盒骨骼拓扑 + 已知 64×64 皮肤模板」，AI 只需把图颜色映射到恒定模板，无需发明 3D 形状。→ 我们若采用「固定刚性骨骼盒 + 命名身体部件 + 恒定体素网格」，同样解锁该类工作流；且体素是 3D 网格，图→体素上色比 MC 6 面贴图更丰富。刚性骨骼盒一箭双雕：既是 T2c 性能解法，也是 AI 角色管线前提。须先定义 2–4 套基础 rig（人形/四足/团块/飞行）作确定性目标。
- **YSM 细化（用户补充）**：用户指出不要"真 MC 皮肤"（纯 2D 贴图、零形状差异），而要 YSM（Yes Steve Model，MC 自定义体素模型 mod）式——有命名骨骼/部件 schema + 每部件可自定义体素形状 + 动画。→ 正确类比应是「固定 schema（部件名/层级/挂点），形状可变」，AI 针对 schema 生成，部件体素形状自由 → "部分差异"。比标准皮肤更贴合我们。下一步优先起草该 rig schema（命名部件+局部变换+每部件体素网格+示例模型文件）。
